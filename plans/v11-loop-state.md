## Frozen Consumer Rules

- `run-manifest.json` is the only artifact discovery source; readers must not guess default filenames.
- Manifest paths must be safe relative paths. Repository/API/frontend should resolve paths relative to the manifest output directory.
- `schema_ref` uses the singular schema filenames currently emitted by manifest metadata, for example `schemas/review-event.schema.json` and `schemas/recompute-marker.schema.json`.
- Non-null `schema_ref` means payload validation is required; `schema_ref = null` means legacy or runtime-specific artifact payload, not schema absence in the manifest contract.
- JSON artifact `record_count` is always `null`; UI counts must read payload fields or arrays. JSONL `record_count` is the count of non-empty lines.
- `scoring-view.json` is a policy-filtered read model. It excludes rejected facts, facts blocked by review, facts missing `reference_scale`, and provider confidence.
- `scoring-view.json` energy facts do not carry `candidate_id`; consumers that need candidate joins must use manifest join keys plus `canonical-evidence.json`.
- `review-events.jsonl` order is semantic; closure uses input order, so API/repository layers must not reorder events before applying closure.
- `recompute-markers.jsonl` `affected_artifacts` values are filenames, not artifact kinds or manifest paths.
- Missing optional conflict/performance artifacts should produce local `unavailable` state for that panel only, not fail the whole run view.

# SpiroSearch V2.1 Office Hour And Joint Spec Review

Date: 2026-07-05
Reviewed plan: `plans/v2-review-and-optimization-plan.md`
Branch state: `feature/spirosearch-baseline`, no commits yet
Scope: plan-only optimization. No V2.1 implementation is included.

## 1. Executive Decision

V2 should not be implemented as a single large rewrite. The current V2 plan correctly identifies major V1 code risks, but it is still too implementation-centric and not strict enough about scientific transfer, evidence conflict, experiment feedback, and CLI contract.

V2.1 should become the baseline for the next implementation round:

1. Lock the scientific and software contracts first.
2. Treat the local Science 2026 paper as a p-i-n AI/interface-stability reference, not direct n-i-p top HTL proof.
3. Separate direct HTL candidates from comparators, interface enablers, and barrier enhancers before ranking.
4. Promote evidence claims, counter-evidence, experiment batches, and decision digests to first-class contracts.
5. Defer large seed expansion and service deployment until validation and role gates are stable.

Recommended mode: **Selective Expansion**. Keep the original V2 traceability and deterministic-report goals, but add the minimum scientific gates needed to avoid false confidence.

## 2. Office Hour Review

### Premises

- The central risk is not that the system misses a promising material; it is that it ranks a material for the wrong role.
- The local Science 2026 paper is valuable for AI-agent design and interface-stability logic, but its device stack is p-i-n. It must not be used as direct evidence that NiOx, SAM, or Al2O3 is a normal n-i-p top-side Spiro replacement.
- A static candidate table cannot satisfy the industrial goal unless failed samples, controls, replicate statistics, and stress protocols return to the model.
- Database and MCP infrastructure should preserve evidence quality. They should not be introduced before the schema prevents incorrect role mixing.

### Narrowest Useful Wedge

The narrowest V2.1 wedge is:

- strict candidate/evidence schema;
- real local PDF trace validation;
- deterministic `decision-digest.json`;
- role-aware ranking gates;
- counter-evidence and transfer-limitation records;
- experiment result schema for future closed-loop ingestion.

This is smaller than "full industrial platform" but strong enough to keep the next implementation scientifically honest.

### Forced Alternatives

| Route | Description | Why Not Primary |
|---|---|---|
| A. Fast V2 code rewrite | Implement all V2 tasks at once. | Too much shared core churn across models, scoring, pipeline, CLI, and data. |
| B. Data expansion first | Expand to 26 or 300 candidates before schema hardening. | More rows would amplify role-mixing and weak evidence problems. |
| C. Infrastructure first | Start PostgreSQL, Neo4j, MCP, MLflow, DVC now. | Service complexity would hide scientific contract gaps. |
| D. Contract-first V2.1 | Lock schema, traceability, digest, role gates, and experiment feedback contract. | Recommended path. |

## 3. Joint Expert Findings

### Materials Science Findings

1. **P0: Architecture transfer is under-specified.**
   Evidence from p-i-n, SAM, Pb-Sn, wide-bandgap, and conventional-bandgap systems must be labeled before it affects n-i-p Spiro replacement ranking.

2. **P0: Evidence labels must gate ranking.**
   `direct_nip_demo`, `nip_hybrid_demo`, `pin_transfer_candidate`, `device_adjacent_evidence`, and `class_prior` should be explicit claim fields.

3. **P0: Counter-evidence is missing.**
   Cu/I migration, SAM desorption, acidic anchors, conductive flake shunts, electrode diffusion, solvent damage, and p-i-n transfer failure must be stored and surfaced.

4. **P1: Architecture pairing is underspecified.**
   SAM, Al2O3, graphene, MXene, MoS2, and h-BN should default to architecture-pairing opportunities unless paired with a base HTL.

### Data And Evidence Findings

1. **P0: Claim schema is not yet a unified evidence event model.**
   It must represent literature, database, calculation, experiment, curation, negative evidence, and derived evidence.

2. **P0: Local PDF trust needs levels.**
   Line anchors are necessary but not sufficient. Record PDF hash, extracted text hash, extraction method, DOI metadata, and anchor status.

3. **P1: `decision-digest.json` needs a formal boundary.**
   Include normalized inputs, schema versions, formula/filter versions, claim IDs, artifact hashes, normalization rules, and ranking config. Exclude timestamps and runtime logs.

4. **P1: Schema validation cannot remain an open decision.**
   V2.1 must define JSON schema files, even if the first implementation validates them with standard-library code.

### Engineering Findings

1. **P0: V2 is too large for one implementation pass.**
   Core files would all change at once: `models.py`, `scoring.py`, `pipeline.py`, CLI, seed data, and tests.

2. **P0: CLI contract is not explicit enough.**
   Define exit codes, stdout/stderr behavior, success artifacts, failure artifacts, and validation error shape.

3. **P0: Trace matching must tolerate PDF extraction variation.**
   Use normalized matching, shorter anchors, optional regex patterns, line spans, and duplicate-match reporting.

4. **P1: Evidence-first scoring should be staged.**
   Keep legacy manual scores as a migration fallback, but emit warnings, penalties, and component evidence maps.

### Experiment And Active-Learning Findings

1. **P0: Experiment schema is missing from the implementation contract.**
   Batch, sample, replicate, comparator, stress protocol, raw data URI, and negative result records are required.

2. **P0: Active learning must be constrained and multi-objective.**
   Acquisition cannot optimize only predicted PCE. It must include stability, manufacturability, uncertainty, diversity, and failure-learning value.

3. **P1: Device claims need comparator and replicate gates.**
   Full-device claims should not be promoted without same-batch Spiro comparator and replicate statistics.

4. **P1: Operational-stability claims need protocol fields.**
   MPPT, ISOS-D/L/O, temperature, humidity, encapsulation, atmosphere, light source, duration, T80/T90/T95, and retained output must be explicit.

## 6. Ranking And Action Rules

### Ranking Eligibility

- `candidate + direct_htl`: eligible for direct ranking only if architecture context and evidence labels pass.
- `candidate + bilayer_htl`: eligible for paired ranking with explicit stack order.
- `comparator + baseline_only`: shown only in baseline/comparator section.
- `interface_component` and `barrier_component`: shown in architecture opportunity section unless paired.
- `architecture_pair`: ranked separately from single-material candidates.

### Recommended Actions

Actions should be derived after filters and evidence review:

- `reject`
- `curate_evidence`
- `calculate`
- `source_or_synthesize`
- `film_screen`
- `half_device_screen`
- `device_screen`
- `stability_screen`
- `architecture_pairing`

Unavailable but promising materials should route to `source_or_synthesize` or `calculate`, not `device_screen`.

### Scorecard Requirements

Every ranked item must include:

- total score;
- component scores;
- component weights;
- uncertainty;
- `component_evidence_map`;
- `component_missing_penalties`;
- top supporting claims;
- top refuting claims;
- conflicts;
- role-specific Pareto dimensions;
- formula version;
- normalization version.

Legacy manual scores may remain in V2.1 only as `legacy_manual_score` fallback and must produce a warning.

## 7. Experiment Gates

### Film Gate

Required checks:

- solvent orthogonality;
- thickness window;
- roughness or pinhole evidence;
- contact angle;
- UV-vis optical loss;
- PL/TRPL or PLQY interface response;
- UPS/KPFM work function or HOMO proxy;
- conductivity or mobility proxy;
- thermal and UV pre-screen.

### Half-Device Gate

Required checks:

- hole-only SCLC or conductivity;
- contact resistance;
- interface recombination proxy;
- compatibility with paired SAM, oxide, or barrier layer;
- no severe perovskite damage from process solvent or temperature.

### Full-Device Gate

Minimum requirements:

- same-batch Spiro-OMeTAD comparator;
- same stack and process window;
- `n >= 6` devices per batch for exploratory screen;
- `n >= 12` devices and at least 2 independent batches for promotion;
- forward and reverse J-V;
- stabilized power output;
- EQE-integrated Jsc;
- hysteresis index;
- median, IQR, best, yield.

### Stability Gate

Required protocol fields:

- ISOS category: `ISOS-D`, `ISOS-L`, `ISOS-O`, or custom;
- MPPT or open-circuit tracking mode;
- temperature;
- humidity;
- atmosphere;
- illumination source and intensity;
- encapsulation;
- duration;
- retained PCE, Voc, Jsc, FF;
- T95, T90, T80 where available.

No claim may be labeled `operational_stability` without protocol fields.

## 8. Active Learning Policy

V2.1 should not automate experimental selection without human review. It should produce a constrained shortlist.

Acquisition must be multi-objective:

- predicted PCE or performance proxy;
- predicted MPPT/ISOS retention;
- manufacturability;
- cost and availability;
- evidence quality;
- uncertainty;
- candidate diversity;
- failure-learning value;
- constraint pass probability.

Failed samples and negative results must enter the training set. The system must distinguish likely material-intrinsic failure from process failure.

### Required Artifacts

- `screening-report.json`
- `screening-report.md`
- `evidence-chain.json`
- `decision-digest.json`
- `run-manifest.json`

### Exit Codes

| Code | Meaning |
|---:|---|
| 0 | Success. |
| 1 | Validation error. |
| 2 | Local paper trace failure. |
| 3 | Input/output path error. |
| 4 | Internal error. |

### Validation Error Shape

Each error must include:

- `error_code`
- `json_pointer`
- `candidate_id`
- `claim_id`
- `message`
- `severity`
- `suggested_fix`

### Decision Digest Boundary

Include:

- normalized candidate records;
- normalized claim records;
- evidence artifact hashes;
- schema versions;
- formula/filter versions;
- normalization rules;
- role-gate config;
- Pareto dimension config;
- ranking sort config;
- local paper trace anchor hashes.

Exclude:

- timestamps;
- absolute local paths;
- operator;
- runtime logs;
- machine-specific environment values.

## 11. Implementation Phases

### Phase 0: Contract Freeze

Deliverables:

- schema files;
- CLI exit codes;
- report artifact contract;
- decision digest spec;
- validation error contract;
- role and material-class enums.

Acceptance:

- malformed booleans such as `"false"` fail validation;
- invalid material class fails with candidate-specific error;
- validation failures write `validation-errors.json`.

### Phase 1: Traceability And Trust

Deliverables:

- local trace validator;
- normalized anchor matching;
- line/span evidence;
- PDF/text hash recording;
- trust-level output.

Acceptance:

- missing `pdf/extracted_text.txt` exits with code 2;
- altered anchor text fails unless `--allow-missing-local-paper` is explicitly set;
- local paper anchors are separated from candidate-specific evidence.

### Phase 2: Role-Aware Ranking

Deliverables:

- `record_type`;
- `replacement_mode`;
- architecture context;
- evidence label gates;
- comparator section;
- architecture opportunity section.

Acceptance:

- Spiro-OMeTAD is never ranked as replacement candidate;
- SAM and 2D barriers do not enter direct HTL ranking without architecture pairing;
- p-i-n evidence receives transfer penalty for n-i-p direct ranking.

### Phase 3: Evidence-First Scorecard

Deliverables:

- `EvidenceClaimV2`;
- `EvidenceLinkV2`;
- component evidence map;
- missing-evidence penalties;
- support/refute/conflict report sections;
- legacy score warnings.

Acceptance:

- every ranked component cites claim IDs or penalty IDs;
- conflicting claims are visible;
- unsupported manual score produces warning and uncertainty penalty.

### Phase 4: Experiment Feedback Contract

Deliverables:

- experiment result schema;
- failure mode enum;
- comparator and replicate gates;
- negative result ingestion fixture;
- stability protocol fields.

Acceptance:

- no comparator means no promoted full-device claim;
- no replicate statistics means no promoted device-screen claim;
- MPPT/ISOS stability claim fails validation without protocol fields;
- negative results appear in evidence chain.

### Phase 5: Seed Expansion

Deliverables:

- 26-record curated taxonomy fixture;
- canonical class coverage;
- direct candidates and architecture components separated;
- counter-evidence where known.

Acceptance:

- at least 5 polymer HTMs;
- at least 6 dopant-free small molecules;
- at least 5 inorganic/hybrid entries;
- at least 4 SAM-derived interface entries;
- at least 6 2D/barrier entries;
- each non-direct record routes to pairing, curation, calculation, or source action.

## 12. Parallelization Strategy

| Lane | Workstream | Modules touched | Depends on |
|---|---|---|---|
| A | Schema and validation contract | `schemas/`, `src/spirosearch/validation.py`, tests | none |
| B | Traceability validator | `src/spirosearch/traceability.py`, CLI, tests | A lightly |
| C | Role and ranking gates | `models.py`, `scoring.py`, tests | A |
| D | Report/digest contract | `pipeline.py`, CLI, tests | A, B, C |
| E | Experiment schema docs/fixtures | `schemas/`, `data/`, docs | A |
| F | Seed expansion | `data/seed_candidates.json` | A, C |

Recommended order:

1. Launch A and B in parallel.
2. Start C after A stabilizes.
3. Start D after B and C.
4. Build E in parallel with C/D because it can be schema-first.
5. Defer F until role gates pass.

Conflict warning: C and D both touch core report objects. Keep them sequential or coordinate through a shared report contract.

## 13. NOT In Scope For V2.1

- Real PostgreSQL/Neo4j deployment: keep contracts and schema alignment only.
- Full MCP server implementation: define minimum tools and response shapes only.
- Automated DFT/MD workflows: keep calculation evidence schema only.
- Fully autonomous active learning: produce shortlist and decision record, not unsupervised lab scheduling.
- Hundreds of generated candidates: wait until 26 curated records pass schema and role gates.
- Replacing all legacy manual scores: retain migration fallback with warnings.

## 14. What Already Exists

- `docs/material-taxonomy.md` already contains useful material classes and evidence labels. V2.1 should promote those labels into schema and ranking gates.
- `docs/database-schema.md` already contains mature PostgreSQL evidence and experiment concepts. V2.1 should borrow the evidence item/link model without deploying the database yet.
- `docs/mcp-resources.md` already defines safe PostgreSQL/Neo4j/MCP boundaries. V2.1 should preserve those as future integration contracts.
- `docs/acceptance-tests.md` already defines deterministic baseline behavior. V2.1 should extend it with role gates, digest boundaries, and experiment feedback checks.
- V1 code already has a small deterministic CLI and report writer. V2.1 should evolve it in stages instead of replacing it wholesale.

## 15. Acceptance Checklist

- `python -m unittest discover -s tests` passes.
- CLI success run writes all five success artifacts.
- Validation failure writes `validation-errors.json`.
- Missing local paper trace exits with code 2.
- Repeated runs produce identical `decision-digest.json`.
- `run-manifest.json` may contain timestamp; digest does not.
- Spiro comparator is in baseline section, not candidate ranking.
- Interface/barrier components are in architecture opportunities unless paired.
- Ranked direct HTL candidates have eligible n-i-p evidence or transfer rationale.
- Every ranked score component maps to claim IDs or penalty IDs.
- Counter-evidence and conflicts are visible in report output.
- Negative experiment results can be represented without data loss.
- Operational-stability claims require stress protocol fields.
- 26-record seed expansion is blocked until schema and role gates pass.
